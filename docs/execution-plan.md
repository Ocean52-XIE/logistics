# 执行计划（2026-03-20 至 2026-04-02）

## 1. 目标与原则

### 1.1 本阶段目标

- 在 2 周内把项目从“可演示原型”推进到“可内部试运行的 MVP 闭环”。
- 跑通员工主链路：登录 -> 学习课程 -> 保存进度 -> 参加考试 -> 查看结果。
- 跑通管理主链路：课程发布 -> 培训指派 -> 报表概览。

### 1.2 执行原则

- 先闭环，后扩展；不在本阶段引入直播、绩效、复杂审批等能力。
- 前后端统一以真实 API 为准，逐步移除前端 prototype/mock 兜底。
- 每个功能点必须同时满足：接口可用 + 页面可用 + 基础测试可回归。

## 2. 范围定义

### 2.1 本阶段必须完成（In Scope）

- 鉴权与权限：员工/管理员访问边界稳定。
- 学习模块：课程列表、课程详情、章节学习、进度自动保存。
- 考试模块：考试详情、自动暂存、交卷幂等、结果展示。
- 管理模块：课程管理（最小可用）、培训计划发布（最小可用）、报表概览。
- 工程能力：测试底座、CI 门禁（typecheck/build/test）。

### 2.2 本阶段暂不纳入（Out of Scope）

- 小程序、移动端深度适配。
- AI 问答助教。
- 与 HR/OA/企业微信集成。
- 复杂规则引擎与多层审批流。

## 4. 验收标准

- 新员工可独立完成 1 条完整学习路径并提交考试。
- 管理员可完成课程发布、培训指派、报表查看。
- 员工不可访问管理端写操作接口，管理员权限边界正确。
- 核心流程具备自动化回归能力，CI 门禁稳定通过。

## 5. 风险与应对

- 风险：需求膨胀导致延期。
  应对：严格按 In Scope 执行，新增需求进入下一阶段池。
- 风险：前端仍混用 mock 与真实数据。
  应对：按页面逐步移除 mock，并在 PR 中明确“数据来源”。
- 风险：考试高并发或网络抖动导致提交异常。
  应对：优先保障暂存、幂等提交、失败可重试。

## 6. 每日执行节奏（建议）

- 10:00 站会：确认当天目标与阻塞。
- 18:00 日结：同步进展、风险、次日计划。
- 每日 merge 前必须通过：`typecheck`、`build`、测试（若已接入）。
